# FitMeet 企业官网桌面端设计审查

> 审查日期：2026-07-11  
> 审查范围：仅桌面端（1440 × 900），不包含移动端设计  
> 审查对象：当前 Next.js 实际路由、页面首屏、主要滚动章节、导航、页脚、核心交互与发布准备度  
> 当前定位：纯静态企业宣传官网；以品牌展示、产品理解、安全信任和 App 上线预告为主，不把 Web Agent 或后台能力纳入完成标准

## 0. 2026-07-11 实施回写

本轮已按审查建议完成四阶段整改。以下状态以当前代码、生产构建和桌面浏览器复测为准。

### 已完成

- 建立统一 `SiteFooter`，已覆盖 Home、World、Moments、Agent、Safety、Support、App、About、Contact、Privacy、Terms、Community Guidelines 与品牌 404。
- World 页脚中的 About、Guidelines、Privacy 已由假入口改为真实链接。
- 新增 `/contact`，提供商务、媒体、安全与隐私公开联系路径，不使用无数据接收端的伪表单。
- 新增 `/support`，提供异常情况第一响应、安全/隐私/规则联系渠道，并明确 App 内举报、屏蔽和申诉仍在建设中。
- 新增品牌化 `app/not-found.tsx`，提供首页、World 与 App 三条返回路径。
- About、Contact、Support、Privacy、Terms、Guidelines 与 404 在导航品牌胶囊中显示正确页面上下文，政策页不再错误高亮 Safety。
- Privacy 和 Terms 已扩展为可供法务继续审核的完整章节结构，包含版本、生效状态、更新时间、信息类别/服务范围、用户权利/责任、处理边界和联系入口。
- Community Guidelines 已增加治理措施结构、举报、申诉、紧急情况、版本与日期。
- 首页保留原 Social World 首屏，但增加 Agent/App CTA，并将重复的四场景长卷缩短为一个 World 入口预览。
- App 与 Agent 的商店入口已由伪下载按钮改为“即将上线”状态和真实 Contact 路径。
- Moments 已强化 `CONCEPT FILM · NOT USER TESTIMONIAL`，明确当前不是用户证言。
- About 已增加当前阶段、合作方向与 Contact 企业信息区；未虚构公司主体、团队规模和办公地点。
- Agent 已增加 Safety 与 App 出口；Safety 已增加 Guidelines、Support 与 Contact 出口。
- Agent 与 Safety 首屏图片改用 Next Image 优化路径，并提升首屏浅色正文对比度。
- 已核对活跃长页面资源：首屏 PNG 由 Next Image 优化，明确保留 `unoptimized` 的章节素材均为约 500 KB 或更小的桌面 JPG；47 MB enterprise 素材库只服务已归档旧首页，不进入活跃页面引用。
- 新增 `sitemap.ts`、`robots.ts`、canonical、Open Graph/Twitter 图片配置与 `.env.example`。
- `metadataBase` 不再回退到 `localhost`；默认使用仓库已登记的 `https://fitmeet.app`，正式部署仍应显式设置 `NEXT_PUBLIC_SITE_URL`。
- 旧 `ArrivalAtlasHomepage` 已移出正式组件目录，归档到 `archive/legacy-home/`；正式首页来源已唯一化。

### 仍依赖外部确认

- Privacy 与 Terms 的结构已经完成，但正式法律效力仍需专业法务审核；页面继续显示 `LEGAL REVIEW PENDING`。
- 公司主体、注册地址、正式隐私联系人和办公地点尚未提供，因此当前没有虚构填充。
- App Store、Google Play 地址、二维码和发布日期尚未验证，因此仍只显示“即将上线”。
- Moments 的真实用户故事仍需要人物与摄影授权；当前定位为品牌概念影像。
- 正式域名已由现有 `.env.example` 确认为 `https://fitmeet.app`；部署环境仍需显式提供 `NEXT_PUBLIC_SITE_URL`，避免不同环境漂移。

- Footer → Contact、Support → First Response、Agent → 生成/确认计划、404 → World 均完成真实交互验证。

- `pnpm -s typecheck` 与 `pnpm -s build` 通过。

## 1. 总体结论

当前官网已经完成了主要品牌页面的视觉设计，不能判断为“整体没做完”。首页、World、Moments、Agent、Safety、App、About 已形成统一且有辨识度的视觉系统：大字号、真实人物场景、轨迹线、空间化滚动和黑白基底上的光谱色都保持一致。

当前主要缺口不是视觉风格，而是企业官网的发布闭环：

1. 访客看完以后缺少稳定、明确的下一步。
2. About、政策页和商务联系入口过深，缺少真正的全站页脚。
3. Privacy 与 Terms 明确仍处于法务占位状态，不能作为正式政策发布。
4. Moments 仍是品牌编辑预告，不是真实内容栏目。
5. 缺少商务合作/联系页面与品牌化 404 页面。
6. 首页与 `/world` 的角色过于接近，信息架构仍有重复。
7. 页面视觉完成度高，但部分小字、浅灰文字和首屏图片加载仍需要上线前优化。

## 2. 现有页面清单

当前实际存在以下正式路由：

```text
/
/world
/moments
/agent
/safety
/app
/about
/privacy
/terms
/community-guidelines
```

因此，既定站点架构里的品牌页与规则页都已有实现。缺失主要发生在企业联系、帮助支持、异常页面和正式发布内容，不是继续增加更多沉浸式展示页。

## 3. 还缺少哪些页面

### P0：商务合作 / Contact

建议新增 `/contact` 或 `/partnerships`。

这是当前最明显的企业官网缺口。现有网站可以让访客理解品牌，但没有一个稳定的页面告诉媒体、合作方、场地方、品牌方或潜在团队成员应该如何联系 FitMeet。

页面至少应包含：

- 商务合作邮箱。
- 媒体与品牌合作邮箱。
- 安全与隐私联系渠道。
- 公司主体名称（准备公开后填写）。
- 可合作方向：运动空间、城市活动、旅行与户外、青年品牌、社区安全。
- 明确回复预期，不使用无法兑现的“24 小时回复”等承诺。

当前仓库中旧的 `ArrivalAtlasHomepage` 仍保留合作章节和 `mailto:`，但当前 `/` 已切换到 `SocialWorldHomepage`，这些内容实际上没有出现在用户路径中。应把有价值的合作信息迁移到正式可访问页面，而不是继续依赖未使用的旧首页组件。

### P0：品牌化 404

建议新增 `app/not-found.tsx`。

当前生产构建会生成框架默认 `_not-found`，但仓库没有品牌化 404。沉浸式官网一旦进入默认错误页，品牌体验会立即中断。

404 页面应保持轻量：

- 一句符合 FitMeet 语气的错误说明。
- 返回 Social World。
- 进入 App 或 Agent。
- 保留全站导航与统一页脚。

### P0：正式 Privacy 与 Terms

路由已存在，但内容仍明确标记为 `LEGAL REVIEW PENDING`，并在正文中说明“不是完整隐私政策/服务条款”。因此它们属于“页面存在，但内容没有完成”，不能作为正式上线的政策文件。

正式发布前需要由法务或合规专业人员完成，不建议只靠视觉与品牌文案替代法律内容。

### P1：Help & Safety Support

建议新增 `/support`，或把完整支持入口并入 `/safety`。

当前 Safety 讲清了品牌原则，但没有解决用户真正遇到问题时“该做什么”：

- 举报与反馈入口。
- 屏蔽、退出和取消计划说明。
- 账号、身份或隐私问题联系渠道。
- 申诉流程。
- 紧急情况提示与平台能力边界。

它不需要做成客服系统，但至少需要提供可执行的静态说明和真实联系渠道。

### P1：News / Updates / Press

建议在有真实内容后新增 `/news` 或 `/press`，当前阶段不必立刻制作空页面。

适合发布：

- App 上线进度。
- 正式产品发布。
- 合作项目。
- 品牌素材包与媒体简介。
- 经验证的活动或社区进展。

没有真实更新时不要先做空栏目。

### P2：Careers

只有在开始公开招聘时再新增 `/careers`。当前阶段不建议为了“看起来像大公司”而制造空职位或虚构团队规模。

## 4. 每个页面存在的问题

### 4.1 首页 `/`

**健康状态：视觉强，定位入口和转化闭环偏弱。**

优点：

- 首屏视觉识别度很高，Social World 的品牌感已经成立。
- 轨迹、碎片图片和真实人物场景形成了可持续的视觉母题。
- 导航、首屏、场景、Agent、安全、结尾的节奏完整。

问题：

1. 首屏只提供“向下进入”，没有清晰的产品型主 CTA。第一次访问的人会先感受到氛围，但不一定知道 FitMeet 到底是什么。
2. 首页和 `/world` 都以 `SOCIAL WORLD` 为核心，首屏内容和场景表达高度相似，两个页面的职责没有完全拉开。
3. 访客要滚动很久以后才能真正理解“辅助型 Agent 如何把需求整理成计划”。
4. 首页结尾只有商店预告与返回顶部，没有 About、Safety、Privacy、Terms、Guidelines、Contact 的完整入口。
5. `APP · COMING SOON` 反复出现，但目前只能提示“即将上线”，没有订阅提醒、联系或可信的后续动作。
6. 首屏部分中文浅灰文字对比度偏低，在亮背景和投影环境下可能难以阅读。

建议：

- 首屏保留品牌主 CTA，同时增加一个明确次 CTA：`了解 FitMeet 如何工作` 或 `查看 App`。
- 把首页定义为“品牌总览”，把 `/world` 定义为“场景地图”，减少重复叙事。
- 用统一全站页脚替代当前极简结尾信息。

### 4.2 Social World `/world`

**健康状态：主体完成，信息层级与出口仍需整理。**

优点：

- 四类场景在一个地图空间里建立了比普通卡片列表更高级的体验。
- 真实照片、分类标签和路线动画相互支持，视觉语言清楚。
- `NOW` 与 `NEXT` 的产品边界基本表达正确，没有把未来能力写成已经上线。

问题：

1. 与首页的首屏名称和场景重合度过高，访客可能不理解“首页”和“World”的区别。
2. 页面末尾视觉上出现 `ABOUT / GUIDELINES / PRIVACY`，但当前实现是普通文字，不是链接，形成明显的假入口。
3. 主要场景 CTA 全部进入 Agent，缺少进入 App、Safety 或具体 Moments 的分流。
4. `NEXT / SOCIAL LIFE` 的愿景较大，最好继续明确“未来方向”而不是让用户误以为已经支持所有生活协作。
5. 页面长度较长，缺少一个始终可见或阶段性的章节位置提示。

建议：

- 把页脚中的假入口改为真实链接，并统一到全站页脚组件。
- 用 Moments 承接真实/编辑故事，用 Agent 承接产品理解，用 App 承接转化，避免所有场景只走一个出口。

### 4.3 Moments `/moments`

**健康状态：视觉完成，内容产品尚未完成。**

优点：

- 编辑式胶片结构有记忆点，与普通社交 Feed 明显区分。
- 页面已经使用 `BRAND EDITORIAL PREVIEW` 和“真实故事需要授权”的提示，避免直接伪装成用户评价。

问题：

1. 它仍然不是一个真正完成的故事栏目；当前全部属于品牌概念影像和编辑预告。
2. “概念影像”提示字号较小、位置偏边缘，普通访客仍可能把照片理解为真实用户案例。
3. 缺少故事署名、地点、时间、授权状态、内容来源等可信信息。
4. 故事之间视觉变化较大，但内容结构相近，尚未形成可持续的编辑模板。
5. 页脚只连接 World、Agent 和 Safety，缺少完整公司与政策入口。

建议：

- 如果短期没有真实授权故事，把栏目名称明确为 `Editorial Preview / Brand Film`。
- 如果准备正式上线 Moments，先建立内容模板与授权规范，再发布第一批真实故事。
- 不要为了填满页面而虚构评论、数字或人物经历。

### 4.4 Agent `/agent`

**健康状态：产品表达正确，首屏可读性和最终出口需优化。**

优点：

- “你负责想去，Agent 负责把条件整理清楚”准确表达了辅助社交定位。
- 时间、地点、兴趣、预算和边界的语义提取非常直观。
- 生成计划与确认计划具有真实前端状态，不是纯装饰界面。
- 明确写出“不替你邀请、不替你承诺、不代替真实相处”，边界清楚。

问题：

1. 首屏浅灰中文和中部需求句对比度偏低，视觉气氛强于信息可读性。
2. 首屏标题、照片和语义轨迹的空间关系略松，右侧留白较大，核心演示没有立即成为第一视觉焦点。
3. 页面最终出口主要是商店预告和返回 World，缺少进入 Safety 的直接路径。
4. 商店按钮仍只显示“即将上线”提示，不能形成转化闭环。
5. Agent 首图触发了 Next.js LCP 加载警告，说明首屏图片加载优先级还需处理。

建议：

- 提高首屏中文说明和需求句对比度。
- 在产品演示完成后增加 `了解安全边界` 与 `查看 App` 两条明确路径。
- 优化首屏图片加载属性和资源大小。

### 4.5 Safety `/safety`

**健康状态：品牌原则完成，实际支持能力未完成。**

优点：

- 没有使用廉价的盾牌、红色警报等通用安全视觉，整体高级且克制。
- Identity、Consent、Public First、Real Experience、Response 五层结构清楚。
- `PRINCIPLE / BUILDING` 能区分原则与建设中能力。

问题：

1. 当前主要讲“我们相信什么”，没有讲用户遇到问题后“具体怎么做”。
2. 缺少举报、屏蔽、退出、申诉、隐私联系和紧急情况说明。
3. 隐私政策仍未完成，Safety 页的信任承诺缺少正式政策支撑。
4. 首屏较浅的小字对比度不足。
5. Safety 首图也触发了 LCP 加载警告。
6. 页面底部只进入 Guidelines 或显示 App 即将上线，缺少 Support/Contact。

建议：

- 保留当前品牌页面，同时增加可执行的支持区或独立 `/support`。
- 只描述已经实现的安全机制；建设中的能力继续明确标记。

### 4.6 App `/app`

**健康状态：视觉与产品演示较完整，转化能力未完成。**

优点：

- 这是全站最清楚的产品转化页，`SAY / SHAPE / CONFIRM / LIVE` 结构容易理解。
- 手机界面和现实场景之间的关系明确。
- 商店未上线时没有伪造下载地址或二维码，这一点正确。

问题：

1. App Store 与 Google Play 外观像可下载按钮，但点击后只是“即将上线”提示，容易造成预期落差。
2. 没有“上线通知”“商务联系”或官方更新渠道，访客无法留下任何意向。
3. 缺少产品阶段、预计发布窗口、支持平台等经过确认的信息。
4. 页尾没有全站公司、政策、安全和联系入口。
5. 产品 Mockup 很有完成度，但应持续避免展示尚未实现的能力为真实可用状态。

建议：

- 未上线阶段把按钮明确写成状态标签，或增加真实可用的“获取上线通知”。
- 如果不接入表单，可以先使用公开邮箱或已验证的官方社交渠道，不应制作无数据去向的伪订阅框。

### 4.7 About `/about`

**健康状态：品牌故事完成，公司可信信息不足。**

优点：

- 从寻找成本、计划和信任切入，使命表达清楚。
- `NOW / NEXT / LATER` 避免了虚构公司年份和规模。
- 页面照片和长线愿景保持了克制，没有过度包装。

问题：

1. About 没有进入顶部全局导航，也没有明确活跃状态，入口过深。
2. 当前更像“品牌宣言页”，不是完整的企业 About 页面。
3. 缺少公司主体、团队/创始人（准备公开时）、所在地、联系渠道、媒体资料与合作方向。
4. 页面末尾只有 World 和 App，没有 Contact、Privacy、Terms 等企业信息入口。
5. `LATER` 愿景较大，需要持续和当前产品阶段清楚分隔。

建议：

- About 保留现有品牌故事，在后半部分增加简洁的企业信息与联系模块。
- 至少让 About 出现在统一页脚中；是否进入顶部导航可以保持当前克制策略。

### 4.8 Privacy `/privacy`

**健康状态：视觉完成，法律内容未完成。**

当前只有最少必要、用途明确、用户控制和待审核说明。正式政策通常还需要：

- 数据控制者/运营主体与联系方式。
- 收集的信息类别与来源。
- 每项信息的使用目的和法律依据。
- 保存期限。
- 第三方共享与处理者。
- 跨境、Cookies/分析工具（如适用）。
- 用户访问、更正、删除、撤回和投诉权利。
- 未成年人规则。
- 安全措施与事件通知。
- 生效日期、更新日期和版本。

设计问题：

- 巨型标题有品牌感，但政策页真正重要的是可扫描、可引用和可检索；正文需要更标准的编号、版本和目录层级。
- 当前顶部导航把 Safety 标记为 active，容易让人误以为仍在 Safety 页面。

### 4.9 Terms `/terms`

**健康状态：视觉完成，法律内容未完成。**

正式条款通常还需要：

- 服务提供主体。
- 适用年龄与资格。
- 服务定义和功能边界。
- 用户账号与真实表达责任。
- 禁止行为。
- 内容与知识产权。
- 暂停、终止和申诉。
- 第三方服务。
- 免责声明、责任限制。
- 争议解决与适用法律。
- 生效日期、版本和联系渠道。

当前四段原则不能替代正式用户协议。

### 4.10 Community Guidelines `/community-guidelines`

**健康状态：基础原则清楚，治理流程未完成。**

优点：

- Real、Consent、Clear Plan、Public First、Respect、Response 六个原则清楚且符合产品定位。
- 阅读目录与锚点交互有效，桌面端正文可读。

问题：

1. 缺少违规等级和可能后果。
2. 缺少举报、证据、处理、通知和申诉流程。
3. 缺少紧急情况与平台责任边界。
4. 缺少生效日期、更新日期和版本。
5. 仍使用 Safety 的导航活跃状态，不利于识别当前位置。
6. 页面底部只串联下一份文档，没有统一企业页脚。

## 5. 全站结构与细节问题

### 5.1 全站页脚没有真正完成

站点架构文档要求 Footer 提供 World、Moments、Agent、Safety、About、App、Community Guidelines、Privacy、Terms，但当前每个页面都使用不同的局部页脚：

- 首页只提供商店预告和 Back to Top。
- World 的 About/Guidelines/Privacy 只是不可点击文字。
- Moments 只提供 World、Agent、Safety。
- Agent 只提供 World 与 Back to Top。
- Safety 只提供 Guidelines、Agent 与 Back to Top。
- App 没有完整公司与政策入口。
- About 只提供 World 与 App。
- 政策页只提供下一份文档与首页。

这是当前最值得优先修复的全站问题。建议创建一个统一 `SiteFooter`，允许每个页面保留自己的沉浸式收尾，但在其后提供一致的企业链接层。

### 5.2 首页与 World 的页面职责重复

建议重新定义：

- 首页：品牌承诺 + 产品如何工作 + 场景预览 + 信任 + App/联系。
- World：完整场景地图与 NOW/NEXT 世界观。

首页不需要再完整复制一次 World 的长场景旅程。

### 5.3 导航位置感不完整

- About 没有顶部入口，也没有 active 状态。
- Privacy、Terms、Guidelines 都把 Safety 标为 active。
- 长页面滚动以后，用户只知道当前一级栏目，不知道所在章节。
- 政策页与品牌页共用相同的浮动胶囊导航，视觉统一，但信息语义不完全准确。

### 5.4 CTA 外观和真实行为不一致

商店按钮外观像真实下载按钮，但目前只显示提示。设计上应明确区分：

- 真正链接：可点击并进入外部商店。
- 即将上线状态：不可下载，但可以获取通知或查看发布进度。
- 纯标签：不要使用强按钮外观。

### 5.5 小字与浅灰文字可读性

首页、Agent、Safety 的浅灰说明文字在大面积亮背景中对比偏弱。英文元信息大量使用极小字号和大字距，视觉精致，但在普通显示器、投影或浏览器缩放下可能难读。

建议在不破坏风格的前提下：

- 提升正文与背景的对比。
- 关键中文正文不低于稳定可读尺寸。
- 极小英文只用于次要元信息，不承载关键含义。

### 5.6 视觉模板重复

多个内页都使用“左侧巨型英文标题 + 右侧场景图/轨迹 + 长滚动章节”。统一性很好，但连续浏览时会出现模板感。

不需要重做视觉系统，只需在页面功能上拉开差异：

- World 强调地图探索。
- Moments 强调编辑阅读。
- Agent 强调真实交互。
- Safety 强调可执行规则。
- App 强调产品序列与转化。
- About 强调企业身份与联系。

### 5.7 页面过长与节奏成本

品牌页桌面高度普遍约为 7,000–9,800 px。沉浸感成立，但每一页都保持相同长度会增加访问成本。

建议：

- 首页缩短，承担总览而不是完整展开所有二级页。
- 二级页只保留本页最独特的体验。
- 提供章节进度或更明确的跳转入口。
- 避免同一事实在首页、World、Agent、App 重复讲述。

### 5.8 首屏性能与图片策略

桌面审查中，Agent 与 Safety 首屏图片触发 Next.js LCP 加载警告。上线前应处理：

- 首屏主图的加载优先级。
- 图片尺寸与实际显示尺寸匹配。
- `unoptimized` 图片的体积控制。
- 长页面懒加载策略。
- 视频 Poster 与实际视频的加载切换。

### 5.9 SEO 与分享细节未完整

当前已有基础 metadata，但未看到以下发布级配置：

- `sitemap.ts`。
- `robots.ts`。
- 品牌化 Open Graph 图片。
- 每个页面的 canonical。
- 正式站点 URL 环境配置校验。
- manifest（如果需要类 App 安装体验）。

全局 `metadataBase` 在未配置环境变量时回退到 `http://localhost:3000`，发布流程必须保证正式环境设置正确。

### 5.10 旧首页代码形成维护歧义

当前 `/` 使用 `SocialWorldHomepage`，但仓库仍保留完整的 `ArrivalAtlasHomepage`、对应 CSS、文案与图片映射。两套首页方向并存会造成：

- 后续维护者不知道哪一套是正式版本。
- 旧版已有的合作 CTA、表单和企业章节不会被用户看到。
- 修改可能落到未使用组件上。

建议确认正式首页方向后，将旧版本归档到明确目录或删除无用实现，并把仍有价值的内容迁移到正式页面。

## 6. 建议实施顺序

### 第一阶段：上线闭环（P0）

1. 建立统一 `SiteFooter`，覆盖全部品牌、公司、规则和联系入口。
2. 新增 Contact/Partnerships 页面，并把 About、Safety、App 接入。
3. 完成品牌化 404。
4. 完成正式 Privacy 与 Terms 内容，并增加版本/日期/联系方式。
5. 修复 World 页脚中的假入口。
6. 明确 App 未上线 CTA 的真实行为。

### 第二阶段：内容可信度（P1）

1. 决定 Moments 是“概念编辑预告”还是“真实故事栏目”。
2. 为 Safety 增加支持、举报和申诉入口。
3. 为 About 增加企业身份、联系与合作信息。
4. 拉开首页与 World 的职责，缩短首页重复内容。

### 第三阶段：精细化（P2）

1. 提高浅灰正文和超小英文的可读性。
2. 优化 Agent、Safety 首图 LCP。
3. 补 sitemap、robots、OG 图片和 canonical。
4. 清理或归档未使用的旧首页实现。
5. 统一政策页导航状态、页脚和文档版本样式。
